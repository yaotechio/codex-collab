针对以下需求，严格按五阶段执行，**不得跳步**：

$ARGUMENTS

**通则**：每次调用 `codex` 都显式传 `cd=<当前项目绝对路径>`，确保 Codex 在正确的项目目录执行、读到该项目自己的规范（如 `AGENTS.md`）。

1. **拆解**：理解需求，**先形成你自己的初步方案与立场**（要点、假设、取舍理由）。这是用来和 Codex 讨论的底稿，不是发给它执行的任务。有歧义或多种解法都摆出来。
2. **讨论**（只读）：用 `codex` 工具、`sandbox=read-only`，**复用同一 `session_id`** 与 Codex 进行**平等的方案辩论**——不是派活。每轮的 PROMPT 必须是「我倾向 X，理由是 Y，你认为哪里有问题/有没有更好的做法？」这类**亮出己见 + 征求异议**的话，而不是「请实现/请分析 X」。拿到 Codex 回复后，你要带着判断回应：认同就说为什么，不认同就反驳，让分歧收敛。方案稳定即提前停止。轮次上限由 MCP server 的 `CODEX_MCP_MAX_ROUNDS` 控制（超限会被熔断拒绝），返回结果里的 `rounds_remaining` 是剩余轮数，据此收敛、不要硬顶上限。
3. **定稿**：综合双方讨论给出《最终方案》，注明哪些采纳了 Codex 的意见、哪些经辩论后保留己见。**停下来等用户确认**，未确认不得进入实现。
4. **实现**（写）：用户确认后，以《最终方案》为 `PROMPT`、**不传 `session_id`（新起干净会话）**、`sandbox=workspace-write`、`confirmed=true` 调 `codex`。在 PROMPT 前置如下给 Codex 的编码规则：
   - 简化优先：解决问题的最小代码，不做未要求的功能/抽象/配置。
   - 外科手术式改动：只改必须改的，不顺手重构无关代码，匹配既有风格。
   - 目标驱动：实现后自检是否满足《最终方案》各条。
5. **验收**（只读）：对照《最终方案》逐项检查 Codex 产物——安全性、业务一致性、覆盖度、是否有越界改动。不达标则回到第 2 步修补。
