# 需求规格：Claude↔Codex 本地协作 MCP Server（Go）

> 状态：已定稿，可直接实现。本文档只含确定要做的部分。

## 1. 目标

构建一个 **Golang 单二进制 MCP Server**，挂载到 **Claude Code**。Claude 拿到用户需求后，自动多轮拉起本机 **Codex CLI** 讨论方案、收敛定稿、派 Codex 实现，最后由 Claude 对照定稿方案验收。

## 2. 架构

- **传输**：stdio + JSON-RPC（MCP 标准）。
- **Server 职责**：极薄。只做「单轮执行 Codex + 会话续接 + 硬熔断限制」三件事。
- **讨论循环由 Claude 驱动**：MCP 工具是请求-响应式，Server 无法反向调用 Claude；多轮讨论靠 Claude 的 agent loop 携带同一 `session_id` 反复调用 `codex` 工具实现。
- **底层调用**：统一执行 `codex exec --json`，从其 stdout 解析结果与会话 id。
- **建议依赖**：MCP SDK 用 `github.com/mark3labs/mcp-go`，不手写 JSON-RPC。

## 3. MCP 工具定义：`codex`

向 Claude 暴露单一工具 `codex`。

### 3.1 入参

| 参数 | 类型 | 必填 | 默认 | 说明 |
|---|---|---|---|---|
| `PROMPT` | string | 是 | — | 发给 Codex 的指令 |
| `cd` | string | 否 | `.` | Codex 工作目录 |
| `sandbox` | string | 否 | `read-only` | `read-only` / `workspace-write` / `danger-full-access` |
| `session_id` | string | 否 | 空 | 传入则续接该会话；不传则新建会话 |
| `skip_git_repo_check` | bool | 否 | false | 允许在非 git 仓库执行 |
| `model` | string | 否 | 空 | 指定 Codex 模型；空则用 codex 默认配置 |
| `return_all_messages` | bool | 否 | false | true 时返回 Codex 完整推理过程与工具调用，便于讨论/调试 |
| `confirmed` | bool | 否 | false | 写操作闸门：`sandbox` 为 `workspace-write`/`danger-full-access` 时必须为 true，否则 Server 拒绝（见第 10 节） |

### 3.2 返回（JSON）

| 字段 | 说明 |
|---|---|
| `success` | 布尔，本轮是否成功，供 Claude 可靠分支 |
| `session_id` | 本轮会话 id（新建时由 Server 从 codex 输出解析回传） |
| `output` | Codex 本轮最终回复 / patch 文本；`return_all_messages=true` 时附完整推理与工具调用事件 |
| `round` | 本会话已进行轮次 |
| `rounds_remaining` | 剩余可用轮次（为 0 即触发硬熔断） |
| `error` | 结构化错误（codex 未装 / 非 0 退出 / 超时 / 超限） |

## 4. 参数 → codex CLI 映射

底层统一调用 `codex exec`：

| 工具参数 | codex CLI |
|---|---|
| `PROMPT` | 走 stdin（避免 argv 长度限制） |
| `cd` | `-C, --cd <DIR>` |
| `sandbox` | `-s, --sandbox <mode>` |
| `session_id`（新建） | `codex exec --json` |
| `session_id`（续接） | `codex exec resume <session_id> --json` |
| `skip_git_repo_check` | `--skip-git-repo-check` |
| `model` | `-m, --model <MODEL>`（空则不传） |
| `return_all_messages` | 不加 flag；控制 Server 返回全部 JSONL 事件还是仅末条 message |

`--json` 输出 JSONL（实测 schema）：`thread.started.thread_id` 即新建会话 id；`item.completed` 中 `item.type=="agent_message"` 的 `text` 即 Codex 最终回复。一次性读完后整体返回（不做流式）。

> 注：`codex exec resume` 不接受 `-s/--sandbox` 与 `-C/--cd`（sandbox/cwd 在建会话时锁定）。故续接调用不传这两项——讨论会话因此天然保持 `read-only`。

## 5. 协作工作流（五阶段，Claude 执行）

1. **拆解**：Claude 理解需求，形成初步方案草案。
2. **讨论**（`read-only`，多轮，≤ `max_rounds`）：Claude 携同一 `session_id` 反复调 `codex` 质疑/迭代，收敛即可提前结束。
3. **定稿 + 用户确认**：Claude 产出《最终方案》，**必须经用户确认**才进入实现。
4. **实现**（`workspace-write`，新会话）：Claude 以《最终方案》为 PROMPT、不传 `session_id` 派 Codex 实现，由 Codex 直接写入文件。
5. **验收**（`read-only`）：Claude 逐项对照《最终方案》审查 Codex 产物（安全性、业务一致性、覆盖度），必要时再起一轮修补。

## 6. 会话管理

| 阶段 | 调用 | 会话 |
|---|---|---|
| 讨论第 1 轮 | `codex exec --json` | 新建，Server 抓回 `session_id` |
| 讨论第 2..N 轮 | `codex exec resume <session_id> --json` | 复用同一会话，保留上下文 |
| 转入实现 | `codex exec --json`（不传 id） | 新起干净会话，仅以定稿方案为输入 |

## 7. 限制与护栏

| 项 | 默认值 | 实现层 | 说明 |
|---|---|---|---|
| `max_rounds` 单会话最大轮次 | 6 | Server 硬熔断 | 内存 `map[session_id]int` 计数，超限拒绝并返回 `error`，防无限互掐烧钱 |
| 单次调用超时 | 300s | Server | `exec.CommandContext` + 超时 cancel，级联杀子进程 |
| 默认 sandbox | `read-only` | Server | 仅实现阶段显式传 `workspace-write` |
| 实现前用户确认 gate | 开启 | 协议层 | 阶段 3 必须用户确认 |
| 会话计数 TTL | 24h | Server | 定期清理过期 `session_id` 计数项，防内存泄漏 |

`max_rounds` / 超时 / TTL 均可通过环境变量配置，给上述默认值。

## 8. 实现约束（硬要求）

- **N1**：`exec.CommandContext` + `context.Context`，收到 Abort 级联 kill codex 子进程，杜绝僵尸进程。
- **N2**：严禁任何内容写入 stdout（污染 JSON-RPC）；调试日志一律走 stderr。
- **N3**：编译为单二进制。
- **N4**：用 MCP SDK 注册工具，不手写 JSON-RPC。
- **N5**：错误处理——codex 未安装 / 非 0 退出 / 不在 git 仓库 / 超时 / 超限，均以 `success=false` + 结构化 `error` 回传 Claude。
- **N6**：并发安全——会话轮次计数 map 用 `sync.Mutex` 保护，支持多个独立 `session_id` 任务并行执行互不干扰。

## 9. 安装到 Claude Code

本工具是 stdio 类型 MCP Server，对 Claude 暴露 `codex` 工具。

1. **编译**：`go build -o codexmcp .` 得到二进制（绝对路径，如 `/Users/liuyao/Documents/github/codexMcp/codexmcp`）。
2. **注册**：
   ```bash
   claude mcp add codex-collab -s user \
     -e CODEX_MCP_MAX_ROUNDS=6 -e CODEX_MCP_TIMEOUT=300 \
     -- /绝对路径/codexmcp
   ```
   `-s user` 全局可用；改 `-s project` 则写入项目 `.mcp.json` 仅本项目可用。`-e` 传第 7 节的可配置项。
3. **验证**：`claude mcp list` 显示 `connected`；Claude Code 内 `/mcp` 可见 `codex` 工具。

## 10. 执行流程的强制保障（三层）

目标：让 Claude 严格按第 5 节五阶段执行。原则——**不可逆的"写文件"步骤用硬保证，流程顺序与质量用软引导**。三层叠加。

### ① Server 硬闸（确定性，防不可逆）
- 默认 `sandbox=read-only`。
- `sandbox` 为 `workspace-write` 或 `danger-full-access` 时，入参 `confirmed` 必须为 `true`，否则 Server 直接拒绝并返回 `success=false` + `error`（不执行）。
- 效果：未经定稿确认的写操作在结构上不可能发生；写只会发生在一次刻意带 `confirmed=true` 的调用上。

### ② Claude Code Hook（确定性，客户端最强拦截）
- 配置 `PreToolUse` hook 匹配本工具（工具名形如 `mcp__codex-collab__codex`）。
- hook 脚本检查入参：当 `sandbox` 为写模式时，返回 `permissionDecision: "ask"` 强制弹出用户确认；用户拒绝即阻断。
- 这是 harness 强制，不依赖模型自觉，连模型抽风也拦得住。配置写入 `settings.json`。

### ③ 协议软引导（一致性，管流程顺序）
- 提供自定义 slash 命令 `/codex-collab <需求>` 作为固定入口，每次注入完整五阶段协议指令，保证从"讨论"开始而非直接实现。
- 同步把简版规则写入 `codex` 工具的 description 与项目 `CLAUDE.md`（常驻）。

### 交付物清单（除 Go 二进制外）
- `PreToolUse` hook 脚本 + `settings.json` 片段（②）。
- `/codex-collab` slash 命令定义（③）。
- `CLAUDE.md` 协议片段（③）。

## 11. 编码规范（Claude 与 Codex 双方遵守）

为降低常见 LLM 编码错误，双方遵守以下规范，**按角色侧重，纯提示词层注入，不改代码**。

四条规范：
- **§1 先想后做**：不臆测、不藏困惑、浮现权衡；多解释清楚再动手，不确定就问。
- **§2 简化优先**：解决问题的最小代码，不做未要求的功能/抽象/配置/防御。
- **§3 外科手术式改动**：只改必须改的；不顺手"优化"无关代码、不重构没坏的东西、匹配既有风格；发现无关死代码只提示不删。
- **§4 目标驱动**：把任务转成可验证目标，定成功标准并循环到验证通过。

角色侧重：

| 规范 | Codex（实现者） | Claude（架构+审查） |
|---|---|---|
| §1 | 次要 | 核心 |
| §2 简化 | **核心** | 验收标尺 |
| §3 外科手术 | **核心** | 验收标尺 |
| §4 目标驱动 | 实现后自检 | **核心**：定标准、对照验收 |

注入位置：
- **Claude**：精简版写入 `/codex-collab` 命令与 `CLAUDE.md`。
- **Codex**：阶段 4 实现 PROMPT 由 Claude 前置一段精简 Codex 规则（重 §2/§3/§4）。
