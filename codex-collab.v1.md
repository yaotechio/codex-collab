针对以下需求，严格按五阶段执行，**不得跳步**：

$ARGUMENTS

**通则**：
- 每次调用 `codex` 都显式传 `cd=<当前项目绝对路径>`，确保 Codex 在正确的项目目录执行、读到该项目自己的规范（如 `AGENTS.md`）。
- **思考过程的展示**：codex 的逐步思考/命令/`🤔` 推理全过程已经在该 `codex` 工具调用的返回里（`process` 字段），用户需要时自行点开那次工具调用查看，完整轨迹也逐行写在返回的 `log_file`。因此 **Claude 不要再把 `process` 轨迹另行贴成代码块**——那会在上下文里复制第二份、徒增消耗；正常只需给出基于讨论的结论/裁决。仅当用户明确要求「展开/贴出思考」时才单独展示。调用进行中 Claude Code 还会通过 progress 通知实时显示当前步骤（取决于客户端版本，不保证）。若需纯实时逐行流式，可改用 Bash 手动跑 `codex exec --json … | <codexmcp 二进制路径> fmt`（read-only 专用）。

1. **拆解**：理解需求，**先形成你自己的初步方案与立场**（要点、假设、取舍理由）。这是用来和 Codex 讨论的底稿，不是发给它执行的任务。有歧义或多种解法都摆出来。
2. **对等辩论**（只读）：将己见底稿交由 Codex 审查，通过对抗发现漏洞并促使方案走向成熟。**必须根据讨论的轮次采取不同的沟通策略，防止无限发散。**

   MCP 参数设定：`sandbox="read-only"`，**设置并复用同一 `SESSION_ID`**；每次调用都显式传 `cd=<当前项目绝对路径>`。

   **Prompt 格式强制要求（区分轮次）：**

   **▶️ 首轮发起 (Round 1)：求独立视角** —— 发送给 Codex 的 PROMPT 必须包含以下结构：
   1. **复述人类需求**：原汁原味地向 Codex 阐述人类用户的原始需求。
   2. **阐述初步方案**：亮明 Claude 自己的初步方案与核心立场。
   3. **征求独立方案**：明确要求 Codex 抛开 Claude 的构思，独立分析人类的原始需求，给出 Codex 视角的底层实现方案及对 Claude 方案的质疑。

   **🔄 后续轮次 (Round 2+)：带着判断回应，持续收敛** —— 拿到 Codex 回复后，你要带着判断回应，**严禁「请实现/请分析 X」之类的单向派活**。PROMPT 必须转为以下结构：
   1. **明确表态与裁决**：认同就说为什么，不认同就反驳，继续让分歧收敛。
   2. **亮出迭代己见**：基于上一轮的裁决结果，向 Codex 展示合并后的「当前迭代版方案」。
   3. **继续征求异议**：针对依然存在分歧的点，必须明确指出驳回了 Codex 的哪些点（有理有据的技术反驳），并结合类似「因此我倾向 X，理由是 Y，你认为哪里有问题/有没有更好的做法？」的话术进行深入论证。

   **🛑 智能收敛与熔断机制：**
   - **方案稳定即提前停止**：一旦双方在核心底层实现上达成共识，必须立即主动终止本阶段讨论，进入定稿步骤。
   - **严禁硬顶轮次**：讨论轮次上限由 MCP server 的 `CODEX_MCP_MAX_ROUNDS` 控制（超限会被系统物理熔断拒绝）。每次调用返回的 JSON 里的 `rounds_remaining` 字段是剩余轮数，必须据此主动收敛、不要硬顶上限。
3. **定稿**：综合双方讨论给出《最终方案》，注明哪些采纳了 Codex 的意见、哪些经辩论后保留己见。**停下来等用户确认**，未确认不得进入实现。
4. **实现**（写）：用户确认后，以《最终方案》为 `PROMPT`、**不传 `session_id`（新起干净会话）**、`sandbox=workspace-write`、`confirmed=true` 调 `codex`。在 PROMPT 前置如下给 Codex 的编码规则：
   - 简化优先：解决问题的最小代码，不做未要求的功能/抽象/配置。
   - 外科手术式改动：只改必须改的，不顺手重构无关代码，匹配既有风格。
   - 目标驱动：实现后自检是否满足《最终方案》各条。
5. **验收**（只读）：对照《最终方案》逐项检查 Codex 产物——安全性、业务一致性、覆盖度、是否有越界改动。不达标时**按粒度回退**：编译错误 / 明显 Bug / 越界重构 → 退第 4 步，带上错误日志要求重写；深层业务逻辑缺陷（方案本身有漏洞）→ 退第 2 步重新辩论修补方案。
