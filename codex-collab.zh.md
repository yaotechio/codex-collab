针对下面的需求，按顺序执行五个阶段。**不得跳步。**

$ARGUMENTS

## 通则（每个阶段都适用）

- **工作目录** —— 每次调用 `codex` 都传 `cd=<项目根绝对路径>`（即你当前的工作目录，不确定就先 `pwd` 取），让 Codex 读到项目自己的 `AGENTS.md`。
- **语言** —— 对用户说中文；发给 `codex` 的每个 PROMPT 用英文写。*但*用户的原始需求始终**原文、不翻译**地带给 Codex（任何轮次、用户用什么语言都照原样）——只有你自己的分析才用英文。数据（文件名、代码、命令、日志、引用文本）保持原样。server 无法强制 Codex 的回复语言；它若用别的语言回，照常读懂并用中文转述。
- **不要复述轨迹** —— 不要在对话中重复贴 Codex 的思考过程，用户需要时自行查看工具调用返回。
- **实时流式** —— 想实时看 Codex 思考，用 Bash 只读跑：`codex exec --json … | <codex-collab 二进制> fmt`（MCP 工具要等跑完才返回）。
- **方案 = 编号标准** —— 《最终方案》是唯一事实源：用中文写成稳定编号 `SC-1、SC-2、…`。英文实现 prompt 只是这些 ID 的一次性翻译（保留 ID）；验收按 ID 对回。

## 1 · 拆解

- 先形成你自己的初步方案与立场（要点、假设、取舍理由），作为与 Codex 辩论的底稿，而非派给它的任务。
- 明确陈述假设。有歧义或多种可行方案时，摆出来；有更简单方案时，说出来。
- 真正不清楚时停下来，不要猜。

## 2 · 辩论 —— 只读

- `sandbox="read-only"`，每次调用都传 `cd`。Round 1 不传 `session_id`（新建会话），复用返回的 `session_id` 跑 Round 2+。
- 按轮次使用不同沟通结构，目标是用辩论让方案成熟并快速收敛。

**Round 1 —— 求独立视角。** PROMPT 必须：
1. 复述需求 —— 用户原文、逐字（不翻译）。
2. 亮出你自己的初步方案与立场。
3. 请 Codex 先抛开你的方案、独立分析原始需求并给出它自己的方案，再对你的方案逐点表态——认同的说明为什么、反驳的给出技术理由，而非一味质疑。

**Round 2+ —— 带判断回应、持续收敛。** 严禁"请实现/分析 X"式单向派活。PROMPT 必须：
1. 表态裁决 —— 对 Codex 的方案逐点表态：认同（说明为什么）或反驳（给出技术理由），让分歧收敛；双方共同认可的点才落进方案。
2. 向 Codex 展示迭代方案 —— 整合 Codex 与 Claude 两方、带上你的观点与本轮迭代的当前版本。
3. 继续收敛分歧点 —— 说明驳回了 Codex 哪些点、为什么。这是双方共同打磨、而非 Codex 独自推演：把你更优的备选方案作为输入贡献进去，取两边思路之长融合，按方案优劣（而非出处）择胜。

**方案由你和 Codex 共同打磨；对接用户的是你，但不是传话筒。** 需求出现真歧义时（无论你自己想到、还是 Codex 反问的），只就"只有用户能拍的决定"用中文、用你自己的话问用户（不转发 Codex 原话）。拿到答复后，把它和 Codex 的疑问放在一起权衡，必要时调整方案，再把更新后的立场带进下一轮和 Codex 对齐。只有用户已明确定下的需求点才可自己答；用户未定下的需求点一律回到用户确认——不要用推断去填需求空白。

**强制收敛**
- 双方在核心做法上达成一致即停止。
- `CODEX_MCP_MAX_ROUNDS` 是轮次上限（超限被 server 物理拒绝）；用返回的 `rounds_remaining` 主动提前收敛，别硬顶。
- **僵局上抛** —— 仅当两条都成立才算真僵局：① 最近一轮双方都没拿出新论据/新事实/新备选，只是重复立场；② 分歧是主观取舍、查代码/文档/测试都定不了（能查证就先查，别上抛）。满足才停下，给用户一份清单——**已达成共识的点 / 仍未定的点 / 需你拍板的点（各附双方观点与理由）**，由用户决策；否则继续辩。
- 简单任务可缩短辩论——是缩短不是跳过：每个阶段照走，尤其写前确认（阶段 3）和写后验收（阶段 5）。

## 3 · 定稿
- 综合讨论，输出中文《最终方案》。
- 写明哪些采纳了 Codex 建议，哪些保留己见。
- 方案必须写成可验证的编号成功标准：`SC-1`、`SC-2` … 不可含糊。
- **停下来等待用户确认，未确认不得进入实现。**
- **用户提异议而非确认时，别自己一个人答 —— 方案是与 Codex 共同打磨的。按粒度回退，回退后按顺序继续往后走（不跳步）：**
  - 措辞微调，或用户此时补定某需求且做法不变 → 自己修订《最终方案》并重新呈交确认（仍在阶段 3）。
  - 异议触及技术做法或业务逻辑 → 退回阶段 2（沿用辩论 `session_id`）与 Codex 重新辩论，再照常走阶段 3 → 4 → 5。

## 4 · 实现 —— 写
- 用户确认后，调用 Codex 实现《最终方案》。
- 参数：**不传 `session_id`**（新会话）、`sandbox=workspace-write`、`confirmed=true`。PROMPT 是已确认 `SC-*` 方案的英文翻译（保留 ID）。
- `PROMPT` 前置编码规则：
  - **简化** —— 解决问题的最小代码；不做未要求的功能/抽象/配置。
  - **外科手术** —— 只改必须改的；不重构无关代码；匹配既有风格。
  - **目标驱动** —— 完成后逐条自检 `SC-*`。

## 5 · 验收 —— 只读

对照《最终方案》通过三镜检查 Codex 产物：
- **目标镜** —— 每个 `SC-*` 是否满足？
- **简化镜** —— 有无未要求的功能/抽象/配置或过度工程？
- **外科镜** —— 每处改动是否都能追溯到方案？有无越界重构？

**不达标按粒度回退，回退后按顺序继续往后走（不跳步）。**
- 编译错误 / 明显 Bug / 越界重构 → 退回阶段 4 带日志重写，再走阶段 5 验收。
- 深层业务逻辑缺陷（方案本身漏洞）→ 退回阶段 2（沿用辩论 `session_id`）重新辩论并修补方案，之后照常走阶段 3 → 4 → 5。
